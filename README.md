# Stateful vs. Packet Filtering Firewalls — Research Code

> **Advanced Network Security (A0347132)**  
> Companion code for the research paper: *"Stateful vs. Packet Filtering Firewalls: A Security Performance Study"*

---

## What This Repo Does

This repository contains the full experimental lab used to generate the empirical results in the research paper. It builds two Docker-based firewall targets — one stateless (packet filter) and one stateful (conntrack) — then subjects both to an **11-vector attack battery** using Nmap and hping3, measuring port filtering effectiveness and scan latency for each attack type.

The results are then fed into a Python visualization suite that generates 4 publication-quality charts for the paper.

## Results Summary

These are the **actual results** generated by this lab:

| Attack Type | Stateless Filtered | Stateful Filtered | Security Gap |
|---|:---:|:---:|:---:|
| Normal SYN | 6/6 | 6/6 | None |
| **Spoof Src Port 80** | **0/6 ❌** | **6/6 ✅** | **+6** |
| **Spoof Src Port 443** | **0/6 ❌** | **6/6 ✅** | **+6** |
| **Spoof Src Port 53** | **0/6 ❌** | **6/6 ✅** | **+6** |
| FIN Scan | 6/6 (ambiguous†) | 6/6 (ambiguous†) | None |
| NULL Scan | 6/6 (ambiguous†) | 6/6 (ambiguous†) | None |
| XMAS Scan | 6/6 (ambiguous†) | 6/6 (ambiguous†) | None |
| Fragment 8-byte | 6/6 | 6/6 | None |
| Fragment 16-byte | 6/6 | 6/6 | None |
| Decoy Scan | 6/6 | 6/6 | None |
| ACK Scan | 6/6 | 6/6 | None |

> **†** Both firewalls produce `open|filtered` for FIN/NULL/XMAS — Nmap reports ambiguity when a host drops packets silently with no RST response. This is expected behavior under a pure `DROP` policy and is documented in the paper (Section IV-C).

**Key finding:** The stateless firewall was **completely bypassed** by all 3 source-port spoofing attacks (370–474 ms, attacker received full port map). The stateful firewall blocked all 3 attacks (1,628–1,831 ms, attacker received no information). The latency gap is Nmap timeout waiting — not processing overhead.

## Usage

### Step 1 — Run the Lab

```bash
chmod +x firewall_lab.sh
sudo ./firewall_lab.sh
```

The script will:
1. Create an isolated Docker bridge network (`fw_arena`)
2. Spin up three Ubuntu containers: `fw_stateless`, `fw_stateful`, `fw_attacker`
3. Configure iptables on each target (see [Firewall Configs](#firewall-configurations) below)
4. Install nmap + hping3 on the attacker container
5. Run all 11 attack vectors against both targets
6. Run a 10-second SYN flood stress test against the stateful firewall
7. Save results to `results/scan_results.csv`
8. Clean up all containers and the network

Expected runtime: **3–7 minutes** depending on your machine.

### Step 2 — Generate Charts

```bash
python3 firewall_plot.py
```

This reads `results/scan_results.csv` and outputs 4 PNG charts to `charts/`.

> **Note:** If you run `firewall_plot.py` without running the lab first, it will use the sample CSV included in `results/` (actual data from the paper's lab run).

## Firewall Configurations

### Stateless (Packet Filter)
```bash
iptables -P INPUT DROP
iptables -A INPUT -p tcp --sport 80  -j ACCEPT   # Trust HTTP source port
iptables -A INPUT -p tcp --sport 443 -j ACCEPT   # Trust HTTPS source port
iptables -A INPUT -p tcp --sport 53  -j ACCEPT   # Trust DNS source port
iptables -A INPUT -i lo -j ACCEPT
```

This simulates a **real-world production misconfiguration** — source-port trust — where inbound traffic from ports associated with trusted services is permitted without connection state validation. This is the specific vulnerability the spoofing attack vectors exploit.

### Stateful (conntrack)
```bash
iptables -P INPUT DROP
iptables -A INPUT -p tcp -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p udp -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -i lo -j ACCEPT
```

This represents **current best practice**: only packets belonging to connections previously initiated from within the protected network are admitted. Source port is irrelevant because no spoofed packet can match an existing conntrack state entry.

|---|---|---|---|
| 1 | Normal SYN | `-sS` | Baseline TCP reconnaissance |

## The CSV Bug (Fixed)

The original version of this script had a subtle bug: `grep -c` output captured via `docker exec` embedded literal newlines into the CSV fields, producing malformed rows like:

```
Stateless,Normal_SYN,6,0
0,0
0,1644
```

**Root cause:** `grep -c` prints `N\n`. When captured across a `docker exec` boundary and written directly to CSV via `echo`, the `\n` becomes a literal newline in the file rather than a field separator.

**Fix:** Count port states using `awk` instead, which returns a clean integer with no trailing newline:

```bash
# ❌ Old (buggy):
FILTERED=$(docker exec attacker nmap ... | grep "/tcp" | grep -c "filtered")
echo "$FW,$SCAN,$FILTERED,..." >> results.csv

# ✅ Fixed:
FILTERED=$(docker exec attacker nmap ... | grep "/tcp" | awk '/filtered/{c++} END{print c+0}')
echo "$FW,$SCAN,$FILTERED,..." >> results.csv
```

The `firewall_plot.py` script also includes an **auto-fix** that detects and corrects the malformed CSV format if you have old output from the unfixed script.

## Reproducing Exactly

To reproduce the exact numbers from the paper:

1. Use Docker on a Linux host (not macOS — macOS Docker networking adds latency variance)
2. Run with no other significant CPU/network load
3. The `ubuntu` base image tag should resolve to Ubuntu 22.04 or 24.04 (both work)
4. hping3 and nmap versions don't affect results (we're testing firewall behavior, not tool versions)

Latency values will vary slightly between runs (~50–200 ms variance) because they reflect Nmap's internal timing and Docker bridge latency. The port state results (filtered/closed/open) are deterministic and will always reproduce exactly.

## Key Findings

1. **Source-port trust is a complete bypass** — all 3 spoofing variants passed through the stateless firewall with 0% filtering, disclosing the full port map in under 500ms.

2. **Stateful conntrack is structurally immune to source-port spoofing** — the positive-match requirement (packet must match an existing session) cannot be satisfied by any forged packet.

3. **FIN/NULL/XMAS produce the same Nmap output on both firewalls** under a pure DROP policy — both drop silently, Nmap reports `open|filtered`. The security mechanisms are different but the external behavior is identical.

4. **Latency is a secondary security indicator** — spoof scans completing in ~400ms means the attacker got RST responses (bypass). The same scan taking ~1700ms means the firewall held and Nmap waited for timeout.

5. **Stateful overhead is 8% on SYN scans** (132ms) — negligible for any non-line-rate environment.
